# Skill：项目讲述校准

> 用于帮木安把项目经历打磨成面试可用的 90 秒讲述版本。

## 触发条件
木安提到"帮我讲这个项目"、"这个项目面试怎么说"、或准备面试时。

## 核心原则
1. **只讲木安能解释清楚的**——面试官追问时能接住，不能只有表面说法
2. **真实 > 好听**——不夸大成果，不虚构数据，不用木安不熟的术语
3. **问题驱动**——先讲发现了什么问题，再讲怎么解决，最后讲结果

## 执行流程

### 1. 信息收集（问木安）
- 这个项目解决什么问题？谁的问题？
- 你具体做了什么？（不是团队做了什么）
- 用了什么技术/工具？为什么选这个？
- 结果是什么？有没有量化数据？
- 你从中学到了什么？

### 2. 套用 STAR-L 结构
- **Situation**：一句话背景（X 场景下存在 Y 问题）
- **Task**：你的角色和目标
- **Action**：你做的关键决策和动作（最多 3 步）
- **Result**：可量化或可感知的结果
- **Learning**：一句话收获（可选，面试官问才说）

### 3. 压缩到 90 秒
- 总字数控制在 250-300 字（中文口语语速约 200 字/分钟）
- 砍掉所有"为了全面性"加的细节
- 每个版本标注预估时长

### 4. 追问模拟
准备 3 个最可能的追问：
- 技术选型类："为什么用 X 不用 Y？"
- 深度类："具体怎么实现的？"
- 反思类："如果重新做你会改什么？"

每个追问给出建议回答骨架，标注木安是否能接住。

### 5. 输出格式
```
## [项目名] 面试讲述版

### 90 秒版本
[完整讲述文本，标注预估时长]

### 追问准备
| 问题 | 建议回答 | 能否接住 |
|------|---------|---------|

### 雷区
- 不要说：
- 如果被问到 X，诚实回答：
```

## 校准规则
- 木安说"我做了 AI 拍照识别"时追问：是你写的模型还是调用的 API？精确到你的真实贡献。
- 项目成果没有数据时，用"可感知的变化"替代，不编数字。
- 讲述超过 2 分钟时强制要求砍内容。
