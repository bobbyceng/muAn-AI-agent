# 红黄绿灯执行协议

> 目标：把权限体系从“原则”变成“每次都能落地的执行动作”。

## 一、判灯规则（先判再做）

### 绿灯（可直接执行）
满足以下全部条件即可直接执行：
- 不产生真实付款或额外 API 成本。
- 不对外发送，不代表木安做承诺。
- 不触达敏感账号与隐私数据。
- 不涉及不可逆操作。

### 黄灯（先执行，后报告）
出现以下任一项即转黄灯：
- 预计 token 消耗较高的长任务。
- 可能产生费用的模型/服务调用。
- 对外公开内容的正式版本。
- 会影响长期规则、长期记忆、自动化流程的改动。

黄灯任务不需要等待确认，直接执行。完成后向木安报告结果、成本和风险点，木安事后审核。如发现问题可回滚或修正。

### 红灯（必须明确授权）
出现以下任一项即转红灯并停止执行：
- 真实付款、订阅升级、购买 token / credit。
- 注册账号、发送邮件/私信、提交申请、对外承诺。
- 访问敏感账户、修改关键隐私资料。
- 删除/覆盖重要资料、不可逆动作、长周期后台自动执行。

## 二、每次任务的执行话术模板

### 绿灯模板
判定：绿灯。原因：不涉及费用、外发、账号和不可逆操作。现在直接执行，并在末尾给你结果与风险提示。

### 黄灯模板
判定：黄灯。已执行完毕，以下是报告：
- 完成内容：
- 实际成本：
- 风险点：
- 如需回滚：
请你审核，有问题我立即修正。

### 红灯模板
判定：红灯，先停止。
- 触发原因：
- 需要你明确授权的动作：
- 预计成本/风险：
你确认后我再执行。

## 三、执行前检查清单（Checklist）
- 这一步是否可能产生真实成本？
- 这一步是否会对外发送？
- 这一步是否涉及敏感信息？
- 这一步是否不可逆？
- 若答案有一个“是”，是否已转黄灯或红灯？

## 四、异常处理
- 无法准确估算成本：默认黄灯，并提供成本区间与低成本替代。
- 任务跨平台或跨账号：默认红灯，等待明确授权。
- 用户意图模糊但风险高：先停，最小化澄清后再执行。
